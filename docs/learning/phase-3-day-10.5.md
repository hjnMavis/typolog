## 1. 테스트 우선(red→green)으로 캐시 계약 고정 (#74)

### 왜 필요한가? (한 줄짜리 수정일수록 테스트가 먼저다)

Day 10이 발견한 V-1은 **딱 한 줄**이다: 제출 완성 `onSuccess`가 `['feed']`를 invalidate하지 않는다. "한 줄이면 그냥 넣으면 되지, 무슨 테스트?"라는 유혹이 가장 큰 순간이다. 하지만 **바로 그럴 때 테스트가 가장 필요하다**. 이유는 두 가지다.

첫째, **한 줄은 눈에 안 보인다.** invalidate 목록에 키 하나가 있고 없고는 코드 리뷰에서 놓치기 쉽고, 나중에 리팩터링하다 조용히 사라져도 아무도 모른다. 회귀가 **소리 없이** 일어난다.

둘째, **1줄 수정의 "계약"은 코드가 아니라 의도다.** "제출 완성은 세 개의 서버 상태(`['submission', id]`·`['my','submissions']`·`['feed']`)를 바꾼다"는 것이 진짜 계약이고, 코드 한 줄은 그 계약의 일부 실현일 뿐이다. 테스트는 이 **계약 전체**를 못박는다.

그래서 순서를 뒤집는다: **먼저 실패하는 테스트를 쓰고(red) → 그것이 실패함을 확인한 뒤 → 한 줄을 넣어 통과시킨다(green).** 이게 red→green이다. 테스트가 먼저 실패해야 "이 테스트가 정말 그 결함을 잡는다"가 증명된다. 통과부터 하면 테스트가 아무것도 검증하지 않아도 초록불이 켜진다(거짓 안심).

### Typolog에서는?

- **테스트(먼저)**: `tests/unit/use-submission.test.ts`가 신규 파일로, 두 케이스를 고정한다 — ① 완성 성공 시 `['submission', id]`·`['my','submissions']`·`['feed']` **세 키 전부** invalidate(:71~75), ② 실패 시 **아무것도** invalidate 안 함(:78~94). 검증 방식은 `vi.spyOn(queryClient, "invalidateQueries")`로 호출된 queryKey 목록을 모아 `toContainEqual`로 대조(:50, :71). 훅의 네트워크·Canvas 배선은 전부 mock이라 **검증 대상은 오직 onSuccess의 invalidate 목록**이다(:21~34).
- **구현(나중)**: `src/hooks/use-submission.ts:62`에 `void queryClient.invalidateQueries({ queryKey: ['feed'] })` 한 줄. 주석(:60~61)이 계약의 근거를 남긴다 — *"피드 멤버십도 바뀐다(내 카드가 새로 등장) … S4 토글과 동일 근거."* 이 "동일 근거"가 Day 9 §1의 **값 vs 멤버십** 기준이다(제출 완성 = 피드에 새 카드 등장 = 멤버십 변화 → invalidate).
- **문서 동기화**: `docs/verification/phase3-integration.md` §2의 "제출 완성(A4)" 행이 `invalidate (멤버십 변화 — V-1 수정, Day 10.5 #74)`로 갱신되고 테스트 파일을 근거로 참조한다(:49).

`['feed']` prefix가 `['feed', challengeId]`를 **포괄**한다는 점도 테스트 주석(:74)이 명시한다 — TanStack Query의 invalidate는 prefix 매칭이라 `['feed']` 하나로 모든 챌린지의 피드 캐시를 무효화한다.

### 비유

수도관에 낡은 밸브 하나를 교체하는 작업. "밸브 하나인데 뭘"이라며 그냥 갈아 끼우면, 나중에 누수가 나도 **어느 밸브가 원인인지** 모른다. 숙련공은 교체 **전에** "이 밸브를 잠그면 3번 방 물이 끊긴다"를 먼저 확인한다(red — 결함이 실재함을 증명). 그리고 새 밸브를 끼운 뒤 "이제 3번 방 물이 정상"임을 같은 방법으로 확인한다(green). 이 "먼저 잠가보기"가 red→green이다.

### 자주 하는 실수

- **한 줄이라 테스트를 건너뛴다.** 한 줄일수록 조용한 회귀 위험이 크다. 계약(어떤 상태가 무효화돼야 하나)을 테스트로 못박는다.
- **구현부터 하고 테스트를 나중에 맞춘다.** 통과하는 테스트를 사후에 쓰면 "결함을 잡는지"가 증명 안 된다. **먼저 실패**시켜야 테스트가 유효함이 증명된다.
- **invalidate 목록을 "값이 바뀌는 캐시"만 챙긴다.** 멤버십이 바뀌는 캐시(제출 완성 → feed)를 빠뜨린다(V-1 그 자체). 값 vs 멤버십으로 전 목록을 점검한다.
- **`['feed', cid]`를 일일이 무효화하려 한다.** prefix `['feed']` 하나면 하위 키를 전부 포괄한다.

## 2. RLS 정책의 CRUD 대칭성 — upsert는 UPDATE다 (#80)

### 왜 필요한가? (INSERT는 되는데 두 번째 저장이 막힌다)

Storage 정책을 짤 때 흔히 "본인 폴더에 **올릴** 수 있게" INSERT 정책과 "본인 파일을 **지울** 수 있게" DELETE 정책만 생각한다. UPDATE는 빠뜨리기 쉽다 — "파일을 수정할 일이 있나?" 싶기 때문이다. 그런데 **같은 경로에 다시 올리는 것(upsert=덮어쓰기)은 INSERT가 아니라 UPDATE다.** Supabase Storage의 `upload(path, ..., { upsert: true })`는 객체가 없으면 INSERT, **있으면 UPDATE** 경로를 탄다. UPDATE 정책이 없으면 **첫 업로드는 성공하는데 두 번째(덮어쓰기)가 RLS에 막힌다.**

이게 왜 치명적이냐면, Typolog의 재시도 전략이 **"전 단계 멱등 — 실패 시 처음부터 재실행"**(게이트 A-(f))이기 때문이다. 이 전략은 콜라주 업로드(A6)가 **덮어쓰기 가능**함을 전제로 한다. A6가 성공한 뒤 다음 단계(A4 완성 전이)가 실패하면, 사용자는 "다시 시도"를 누르고 → 오케스트레이터는 처음부터 다시 돌며 → A6를 **같은 path에 재업로드**(=UPDATE)한다. UPDATE 정책이 없으면 이 재업로드가 막히고, 결과적으로 **A6 성공 이후의 어떤 실패든 재시도가 영구 불능**이 된다. 정책 하나 누락이 재시도 전략 전체를 무력화한다.

핵심 감각: **RLS 정책은 앱이 실제로 수행하는 CRUD 연산과 대칭이어야 한다.** upsert를 하면 INSERT + UPDATE 정책이 **둘 다** 필요하다(이건 Phase 2 Day 3에서 이미 배운 원칙인데, collages 버킷에서 UPDATE만 조용히 빠져 있었다).

### Typolog에서는?

- **결함**: `collages` 버킷에 INSERT/DELETE/SELECT 정책은 있으나 **UPDATE 정책이 없었다.** 대조군: `letter-pieces`에는 `letter_pieces_update`("본인만 덮어쓰기 — 글자 교체", 0003 §5.1)가 처음부터 있었다. 글자 교체가 명시적 기능이라 UPDATE를 챙겼지만, 콜라주는 "한 번 만들면 끝"이라는 착각으로 빠뜨린 것이다.
- **수정**: `src/db/migrations/0006_collages_allow_owner_update.sql`이 `collages_update`를 추가한다(:12~18) — `letter_pieces_update`와 **동일 패턴**: `FOR UPDATE TO authenticated USING (bucket_id='collages' AND (storage.foldername(name))[1] = (SELECT auth.uid())::TEXT)`. 경로 첫 폴더 = 본인 uid라는 소유권 표현도 그대로다.
- **발견 경위**: 이 결함은 코드 리뷰가 아니라 **#50 프로파일링 중 실측**으로 잡혔다. `scripts/profile-collage-upload.ts`가 콜드 업로드(새 객체=INSERT)는 성공하는데 **같은 path 덮어쓰기(upsert=UPDATE)에서 실패**하는 것을 관찰했다(:183~190). 마이그레이션 0006 주석(:10)이 이 경위를 남긴다.
- **회귀 프로브 2종(허용+차단 짝)**: `scripts/verify-rls.ts`에 ① A가 본인 콜라주 같은 path 재업로드 **허용**(:483~488), ② B가 A의 콜라주 덮어쓰기 **차단**(:490~495)을 추가했다. "허용" 하나만 넣으면 정책이 너무 넓게 열렸는지(타인도 덮어쓰나?)를 못 잡으므로, Phase 2 Day 5에서 배운 **allow/deny 짝** 원칙대로 음성 케이스를 반드시 붙인다.
- **verify-rls가 왜 이 결함을 못 잡았나**: 기존 Storage 프로브는 **매 실행 새 경로(randomUUID)만 업로드**해 덮어쓰기 경로가 한 번도 발동되지 않았다(verify-rls.ts:480~482 주석). "매번 새 fixture"라는 자체 정리형 스크립트의 안전 습관이 역설적으로 **덮어쓰기 경로를 영영 안 밟게** 만든 것 — 검증의 사각지대다.

### 비유

아파트 각 세대에 "물건을 **들여놓을** 권한(INSERT)"과 "**버릴** 권한(DELETE)"만 주고, "**기존 물건을 새 걸로 교체할** 권한(UPDATE)"을 안 줬다. 처음 이사(첫 업로드)는 되는데, 낡은 소파를 같은 자리에 새 소파로 바꾸려니(덮어쓰기) 경비가 막는다. "버리고 새로 들여놓으면 되잖아"? 그런데 우리 재시도 규칙은 "그냥 같은 자리에 새 걸 놓는다(upsert)"라서, 교체 권한이 없으면 **이사 온 뒤 가구를 영영 못 바꾸는** 집이 된다.

### 자주 하는 실수

- **upsert에 INSERT 정책만 챙긴다.** upsert는 객체가 있으면 UPDATE 경로 → UPDATE 정책도 필요(INSERT+UPDATE 대칭).
- **허용 프로브만 추가한다.** 정책이 과도하게 넓게 열렸는지는 **타인 차단(음성)** 프로브가 있어야 잡힌다. 허용+차단 짝으로.
- **"매번 새 경로"만 테스트한다.** 자체 정리형 스크립트가 새 fixture만 쓰면 덮어쓰기 경로를 영영 안 밟는다. 회귀 프로브는 **같은 path 재업로드**를 의도적으로 유발해야 한다.
- **재시도 전략의 전제를 명시 안 한다.** "실패 시 처음부터 재실행"은 **전 단계 멱등(덮어쓰기 가능)** 을 전제한다 — 그 전제를 깨는 정책 누락은 전략 전체를 무너뜨린다.

## 3. 단일 표본 결론 금지의 실전 — 반복 측정 + 구성 요소 분해 (#50)

### 왜 필요한가? (10.6초짜리 병목을 향해 곧장 달려가면 안 되는 이유)

Day 10 성능 측정에서 A6 콜라주 업로드가 **10,560ms**로 나왔다. 제출 체감 시간을 압도하는 값이라 "여기가 병목이다, 최적화하자"가 자연스럽다. 하지만 Day 10 노트(§5)는 이걸 `⚠️ 지배 비용 — 재측정 필요(단일 표본)`으로 못박고 **결론을 유보**했다. 왜? **딱 한 번 잰 값은 사실이 아니라 가설**이기 때문이다. 첫 요청 웜업, 네트워크 순간 지연, dev 서버의 on-demand 컴파일 같은 일회성 요인이 섞였을 수 있다.

이 유보를 **실전으로 해소하는 방법**이 두 가지다:

1. **반복 측정** — 한 번이 아니라 콜드 1회 + 웜 N회를 재서 분포(p50/p95)를 본다. 이상치는 반복하면 사라진다.
2. **구성 요소 분해** — "A6가 느리다"는 뭉뚱그린 관찰이다. A6는 **인증 왕복 + Storage 업로드 왕복 + DB UPDATE + signed URL 발급**의 합이다. 각 조각을 따로 재야 "무엇이" 느린지 안다(왕복 지배 vs 실행 지배 — Day 10 §5).

이 두 가지가 **병렬화 착수 여부를 결정**한다. 만약 A6가 진짜 수 초라면 병렬화(글자 업로드 동시 실행, 기대효과 ~0.5s)는 **헛수고**다 — 지배 비용은 A6이므로 A6 원인부터 잡아야 한다. 재측정 없이 병렬화부터 착수하면 우선순위가 뒤집힌 채 힘을 쓴다.

### Typolog에서는?

`scripts/profile-collage-upload.ts`가 이 두 방법을 코드로 구현한다:

- **자체 정리형 재측정**(verify-rls와 동일 안전 등급): 테스트 계정 생성 → draft 제출 fixture → 실 JWT 로그인 → 측정 → `finally`에서 Storage 객체·제출 행·계정 순 삭제(:227~234). 쓰기를 하지만 **순 변화 0**. secret은 presence boolean만(:45~50). Day 10에서 A6를 못 다시 잰 이유("본계정·보조계정 모두 오늘 제출이 completed라 라우트 반복 호출 불가", :6~8)까지 스크립트 서두가 설명한다.
- **구성 요소 분해**: 인증 왕복(`auth.getUser`, :150~158), Storage 업로드 콜드 1회 + 웜 N회(:174~181), DB UPDATE(:193~202), signed URL 발급(:205~216)을 **각각** 반복 측정해 `summarize`로 min/p50/p95/max 출력.
- **판정 결과**: 웜 업로드 **p50 164ms** — Day 10의 10,560ms와 **40~60배 괴리** → A6는 **이상치로 판명**(문서 §5, 백엔드 §9 Day10.5-(c)). 인증 34.7ms / Storage 웜 164ms / DB 24ms / 서명 35.5ms → 합계 ~260ms. 유력 원인은 dev 라우트의 첫 on-demand 컴파일(A5 글자1의 콜드 3,995ms와 같은 패턴). 스크립트 말미(:218~226)가 `p50 >= 3000ms`면 "A6 원인 대응 우선", 아니면 "글자 병렬화가 남은 개선"이라는 **판정 가이드**를 직접 출력한다.
- **결정으로 이어짐**: 이상치 판명 → 병렬화 착수 조건(게이트 A ②b) 충족 → §4의 워커 풀 구현으로 진행. **측정이 결정을 만든** 것이다.

### 비유

병원에서 혈압이 딱 한 번 **180**으로 나왔다고 곧장 약을 처방하지 않는다. "방금 계단 뛰어 올라왔나요?"(콜드), "커프가 제대로 감겼나요?"(측정 오류)를 따지고, **여러 번 다시 잰다**(반복). 그리고 "수축기가 높나 이완기가 높나"(구성 요소 분해)를 나눈다. 재보니 안정 시 120이면 그 180은 **이상치**였다 — 약(병렬화)이 아니라 다른 처방(콜드 원인)이 필요하다.

### 자주 하는 실수

- **단일 표본으로 최적화를 착수한다.** 10.6s 한 번 보고 병렬화에 착수 → 실제 지배 비용을 놓치고 헛수고. 반복 측정 전 결론 금지.
- **"A6가 느리다"에서 멈춘다.** 인증/전송/DB/서명 중 무엇인지 분해 안 하면 처방을 못 정한다.
- **재측정을 라이브 데이터 오염으로 한다.** 오늘 제출이 completed면 라우트를 못 부른다 → 테스트 계정 + 자체 정리형으로 순 변화 0.
- **판정 기준을 사후에 정한다.** "몇 ms 이상이면 A6 원인"인지 **측정 전에** 임계값을 정해 둬야 결과에 끌려가지 않는다(스크립트가 3000ms 임계를 코드에 박아둔 이유).

## 4. 워커 풀 동시성 cap + 실패분만 재시도 (#50)

### 왜 필요한가? (전부 동시에 쏘면 회선이 막히고, 하나 실패했다고 전부 다시 하면 낭비다)

글자 업로드(A5)는 서로 독립이다(slot_index가 다르고 각각 UPSERT). 순차로 하면 6~7장이면 초 단위로 쌓인다. 그럼 "전부 `Promise.all`로 동시에!" 하고 싶지만 — 모바일 회선에서 7개를 한꺼번에 쏘면 **서로 대역폭을 뺏어** 오히려 느려지고 Storage에도 부담이다. 그래서 **동시성 상한(cap)** 이 필요하다: "최대 3개까지만 동시에, 하나 끝나면 다음 것."

그리고 병렬 업로드 중 **일부가 실패**할 수 있다. 여기서 순진한 선택은 "하나라도 실패하면 전체 실패 → 처음부터"다. 하지만 이미 성공한 5장을 버리고 다시 올리는 건 낭비다. 각 A5가 **UPSERT 멱등**이므로, **실패한 것만 골라 1회 재시도**하는 게 옳다. 재시도까지 실패하면 그때 에러를 전파해 **전체 재실행(기존 안전망)** 에 맡긴다.

또 하나: 병렬이 되면 **진행 표시의 의미가 바뀐다.** 순차일 땐 "3/5번째 업로드 중"이 자연스럽지만, 병렬에선 3·5·2번이 동시에 진행되므로 "순번"이 무의미하다. 대신 **완료 누적 수(k/N)** 로 바꿔야 한다.

### Typolog에서는? (세마포어 없는 워커 풀)

`src/features/compose/submit-collage.ts`의 `uploadLettersConcurrently`(:94~127):

```
const workers = Array.from({ length: Math.min(cap, queue.length) }, async () => {
  let letter
  while ((letter = queue.shift()) !== undefined) { ...업로드... }
})
await Promise.all(workers)
```

- **cap으로 상한**: `LETTER_UPLOAD_CONCURRENCY = 3`(:71). cap개의 워커(async 함수)를 만들고, **모두 같은 `queue`를 공유**한다(:101~104). 각 워커는 큐가 빌 때까지 `queue.shift()`로 하나씩 꺼내 처리한다. 워커가 3개뿐이므로 **동시 진행은 항상 최대 3개**. 세마포어·카운터 없이 "공유 큐 소비"만으로 상한이 걸린다.
- **`queue.shift()`가 왜 안전한가**: JavaScript는 **단일 스레드**이고 `await` 지점에서만 다른 태스크로 전환된다. `queue.shift()`는 동기 연산이라 **중간에 끼어들 틈이 없다(원자적)**. 두 워커가 같은 letter를 꺼내는 경합이 원천적으로 불가능하다 — 멀티스레드 언어라면 락이 필요하지만 JS에선 불필요하다.
- **실패분만 재시도**: `runBatch(letters, collectFailures=true)`가 1차 배치에서 실패를 던지지 않고 `failed[]`에 모은다(:112~115). 실패가 있으면 `runBatch(failed, collectFailures=false)`로 실패분만 재시도하고, 여기선 실패 시 **즉시 throw**해 전파한다(:123~125). 재시도 성공분도 `onLetterDone`이 클로저로 그대로 호출돼 진행 카운트에 반영된다(:111).
- **진행 표시 전환**: `SubmitProgress.current`가 순번이 아니라 **완료 누적 수(0→N)**(:26 주석, :145~154). UI(`CollagePreviewClient.tsx:39~40`)도 `글자 N장 업로드 중… (k/N)`로 바뀌었다.
- **테스트로 고정**: `tests/unit/submit-collage.test.ts`가 ① 동시 in-flight ≤ 3이면서 실제 병렬(maxInFlight > 1, :107~130), ② 진행 콜백 0→N 단조 증가(:132~150), ③ 부분 실패→재시도 성공 시 호출 4회(3+재시도1, :171~195), ④ 재시도까지 실패 시 에러 전파 + 후속 단계(collage/complete) 미실행(:197~218)을 검증한다.

`Promise.all(workers)`가 각 워커 프라미스에 핸들러를 붙이므로, 하나가 reject해도 **나머지 워커의 미처리(unhandled) rejection이 안 생긴다** — QA가 코드 추적으로 확인한 미묘한 안전성이다.

### 비유

식당에 주문이 20개 밀렸다. 주방장 1명이 순차로 하면 느리고(순차), 요리사 20명을 한꺼번에 투입하면 주방이 아수라장이다(무제한 병렬). 그래서 **요리사 3명(cap)** 을 두고, 각자 **주문 티켓 뭉치(공유 큐)** 에서 한 장씩 뽑아 만든다 — 3명이 쉬지 않으니 항상 3개가 동시에 조리된다. 그리고 3번 주문이 타서 실패하면 **20개 전부 다시 하지 않고** 3번만 다시 만든다(실패분 재시도). 티켓을 뽑는 순간은 찰나라 두 요리사가 같은 티켓을 뽑을 일이 없다(원자적 shift).

### 자주 하는 실수

- **`Promise.all`로 전부 동시 발사.** 모바일 회선·Storage 과부하로 오히려 느려진다. cap을 둔다.
- **하나 실패했다고 전체 재실행.** UPSERT 멱등이면 실패분만 재시도가 낭비 없다. 단, 재시도도 실패하면 전파해 전체 재실행 안전망에 맡긴다.
- **병렬인데 순번(1/N)으로 진행 표시.** 병렬에선 순번이 무의미 → 완료 누적(k/N).
- **워커 풀에 락을 넣으려 한다.** JS 단일 스레드에서 `queue.shift()`는 원자적 — 세마포어·뮤텍스 불필요. (멀티스레드 언어의 습관을 그대로 옮기면 과설계)
- **reject된 워커의 unhandled rejection을 방치.** `Promise.all`이 모든 워커를 대기하므로 처리되지만, 워커 밖에서 fire-and-forget하면 샌다.

## 5. UNIQUE + onConflictDoNothing + returning()의 경합 안전 멱등 (#48)

### 왜 필요한가? ("확인하고 넣기"에는 시간의 틈이 있다)

같은 사람이 같은 글을 두 번 신고하는 걸 막고 싶다. 순진한 방법:

```
const existing = await select(...where reporter=X and submission=Y)  // ① 확인
if (existing) return ALREADY                                          // ②
await insert(...)                                                     // ③ 삽입
```

이 "SELECT-후-INSERT"에는 **①과 ③ 사이에 시간의 틈**이 있다. 사용자가 신고 버튼을 빠르게 두 번 누르거나(동시 요청 2개), 두 요청이 **둘 다 ①에서 "없음"을 보고** ③으로 넘어가면 **중복 2건이 삽입**된다. 이 "확인 시점과 사용 시점의 불일치"를 **TOCTOU(Time-Of-Check to Time-Of-Use)** 라 한다.

해법은 **판정을 앱이 아니라 DB 제약에 맡기는 것**이다. `UNIQUE(reporter_id, submission_id)` 제약을 걸면 DB가 **삽입을 직렬화**해 두 번째를 원천 거부한다(에러 코드 23505). 그리고 `onConflictDoNothing` + `returning()`을 쓰면 "충돌하면 조용히 무시하고 **0행 반환**" → 반환이 0행이면 "중복이었다"를 안다. **확인과 삽입이 하나의 원자적 연산**이 되어 틈이 사라진다. (이건 Day 7 좋아요 토글의 `UNIQUE(user_id, submission_id)` + `onConflictDoNothing` 패턴과 같은 틀 — 신고에 그대로 재사용.)

한 가지 더: **제약을 걸기 전에 기존 중복을 정리**해야 한다. UNIQUE 제약은 위반 데이터가 이미 있으면 **추가 자체가 실패**한다. 그리고 이 정리 DELETE와 제약 추가를 **같은 마이그레이션(트랜잭션)** 에 넣어야, 그 사이에 새 중복이 끼는 경합 창이 없다.

### Typolog에서는?

- **마이그레이션**: `src/db/migrations/0005_same_phil_sheldon.sql`이 ① 결정적 선정리 DELETE → ② `ADD CONSTRAINT UNIQUE`를 **같은 파일(원자적 트랜잭션)** 에 담는다(:9~16). 선정리 규칙은 **결정적**이다: 같은 (reporter, submission) 쌍에서 `created_at` 오름차순, 동시각이면 `id` 오름차순으로 **가장 오래된 행만 남긴다**(:5~6, :13~14). 이 `(created_at, id)` 전순서 타이브레이크가 **어떤 행을 남길지**를 결정적으로 만든다(Day 6 커서 페이지네이션의 전순서 감각과 동형). Day 10 실측: 총 3건 중 중복 쌍 1개(2건) → 1행 삭제.
- **런타임**: `src/lib/actions/reports.ts`의 `createReport`가 `.onConflictDoNothing({ target: [reports.reporter_id, reports.submission_id] }).returning({ id })`(:59~67) → `inserted.length === 0`이면 `REPORT_ALREADY_EXISTS` 반환(:69~71). 주석(:57~58)이 못박는다: *"'확인 후 삽입'과 달리 동시 요청 경합에서도 정확히 1건만 적재된다."*
- **에러 전달 방식**: 결과를 throw가 아니라 **구조화 객체 `{ ok, code }`** 로 반환한다(:13~18). Day 7에서 배운 대로 Next.js가 production에서 throw 메시지를 마스킹하므로, 다이얼로그가 사유별 메시지를 보이려면 반환값으로 분기해야 한다. `CreateReportResult` 유니언에 `REPORT_ALREADY_EXISTS`를 추가하니 `ReportDialog.tsx`의 `resultMessage` **exhaustive switch**(:25~40)가 컴파일 타임에 누락을 강제한다("이미 신고한 글이에요." 매핑, :37~38) — 타입이 UI 갱신을 잊지 않게 지킨다.
- **회귀 프로브**: `verify-rls.ts:378~386`이 같은 (reporter, submission) 2회 INSERT → 둘째가 **23505**여야 함을 라이브에서 확인(첫 INSERT는 본인 신고 정상 경로 WITH CHECK 통과). 라이브 PASS로 마이그레이션 0005 적용까지 실증.

### 비유

콘서트 좌석 예약. "이 자리 비었나 확인 → 예약" 두 단계로 하면, 두 사람이 **동시에 "비었네"를 보고** 둘 다 예약 버튼을 눌러 **한 자리에 두 명**이 배정된다(TOCTOU). 해법은 좌석에 **"한 번만 배정" 잠금(UNIQUE)** 을 걸어, 예약 요청 자체를 DB가 줄 세우는 것 — 두 번째 요청은 "이미 찼습니다"로 튕긴다(23505). 확인과 예약이 **한 동작**이 되니 틈이 없다. 그리고 잠금을 걸기 전, 실수로 이미 이중 배정된 자리는 **먼저 정리**해야 잠금이 걸린다(선정리 DELETE).

### 자주 하는 실수

- **SELECT-후-INSERT로 중복을 막는다.** 두 단계 사이 TOCTOU 틈으로 동시 요청이 중복 삽입. 제약으로 직렬화한다.
- **제약만 추가하고 기존 중복을 안 지운다.** 위반 데이터가 있으면 `ADD CONSTRAINT`가 실패. 선정리 DELETE를 **같은 트랜잭션**에.
- **선정리 규칙이 비결정적.** "아무거나 하나 남김"이면 어떤 행이 남을지 모른다. `(created_at, id)` 전순서로 결정적으로.
- **새 에러 코드를 UI 매핑에 안 반영.** exhaustive switch(유니언 타입)로 컴파일러가 누락을 잡게 한다.
- **충돌을 throw로 알린다.** production에서 마스킹돼 사유별 메시지가 안 뜬다 → `{ ok, code }` 반환.

## 6. 화면 간 정합성의 마무리 — 진입로 차단 + 잔존 데이터 정리 (#78)

### 왜 필요한가? (한 화면만 고치면 다른 화면이 여전히 거짓말을 한다)

Day 10 §9가 진단한 F-3: 서버는 "완성 = 확정(재편집 불가)"인데, **미리보기 화면(#72)** 은 이 확정을 알아 "이미 완성한 콜라주예요"를 복원하는 반면, **수집 화면(#78)** 은 확정을 몰라 로컬 슬롯이 편집되는 것처럼 보였다. 사용자에겐 "편집했는데 미리보기에서 무시되는" 모순이다. Day 10 §9의 처방은 **"한 상태(완성)를 소비하는 모든 화면을 열거하라"** 였다.

Day 10.5는 이 처방을 실행하되, 수집 화면이 "completed임을 알고 편집 UI를 비활성화"하는 방향이 아니라 **더 근본적인 두 수를 둔다**:

1. **진입로 자체를 차단**한다. 완성된 챌린지의 수집 화면(`/challenge/[id]`)에 들어오면 **편집 UI를 보여주지 않고 곧장 확정 화면(미리보기)으로 리다이렉트**한다. "완성된 슬롯이 잠깐이라도 편집 가능해 보이는" 순간(F-3의 뿌리)을 원천 제거.
2. **혼란의 뿌리인 잔존 데이터를 정리**한다. 확정 후에도 남아 있는 **로컬 슬롯 메타 + IndexedDB Blob** 이 "편집되는 것처럼 보이는" 착시의 실체다. 확정 화면 진입 시 이걸 지운다.

그리고 두 화면이 **같은 서버 권위 소스**를 재사용해야 한다 — `#72`가 쓰던 `['my','submissions']`(completed만 반환)를 수집 화면도 그대로 읽는다. 단일 소스라야 두 화면이 절대 어긋나지 않는다(Day 9 §2 단일 소스의 연장).

### Typolog에서는? (짝을 이루는 두 수정)

- **진입로 차단** — `src/features/challenge/CaptureClient.tsx:31~53`: `useMySubmissions()`로 오늘 챌린지의 완성 여부를 조회(:36~38) → 완성이면 `router.replace(preview)`(:40~42). 완성 여부 확인 전(`isMyListPending`)이나 완성 시엔 `TodayChallengeGate`(=`CaptureView`)를 **아예 마운트하지 않아**(:47~53) 편집 UI가 한순간도 안 뜬다. 조회 실패 시엔 수집 화면으로 진행하는 **fail-open**(:46) — 서버가 재제출을 멱등 차단(#60-B)하므로 안전하다. 주석(:33~35)이 `#72와 같은 서버 권위 소스(['my','submissions']) 재사용`을 명시.
- **잔존 정리** — `src/features/compose/CollagePreviewClient.tsx:112~129`: 확정 상태 진입(`!submittedId && completedItem !== null`, :116)이면 로컬 슬롯 메타를 `store.resetDraft()`로 **먼저 비우고**(:127), IDB Blob을 `deleteImageBlobs(keys)`로 정리(fire-and-forget, :128). `store.challengeId !== challenge.id` 가드(:120)로 다른 챌린지 draft는 안 건드린다. "메타를 먼저 비워 stale 편집 진입을 즉시 차단, Blob 삭제는 실패해도 화면 노출 없음"(:125~126)이 `handleResetDraft`와 동일 방침이다. 방금 제출한 화면(`submittedId` 있음)은 콜라주 카드가 로컬 URL을 계속 쓰므로 **제외**하고 다음 진입 때 정리(:114~115).

두 수정이 **짝**이다: 진입로 차단(수집)이 "편집 화면을 안 보여주고", 잔존 정리(미리보기)가 "착시의 재료를 없앤다". 하나만 하면 반쪽이다.

### 비유

"이미 완공된 집"에 인테리어 시공팀이 또 들어오려 한다. 방법 1: 현관에서 **"이 집은 완공됐습니다, 준공식장으로 가세요"라며 돌려보낸다**(진입로 차단). 방법 2: 시공팀이 놓고 간 **자재와 도면(로컬 draft)을 치운다** — 안 치우면 다음에 또 "아직 공사 중인가?" 착각한다(잔존 정리). 둘 다 해야 "완공됐는데 왜 공사판이지?"라는 혼란이 사라진다.

### 자주 하는 실수

- **화면 하나만 고치고 끝낸다.** #72(미리보기)만 고치면 수집 화면이 여전히 거짓말. **소비 화면 전체를 열거**해 짝으로 고친다.
- **상태만 읽고 잔존 데이터를 안 지운다.** "편집처럼 보이는" 착시의 실체는 로컬 슬롯·IDB Blob. 뿌리를 정리해야 재발이 없다.
- **화면마다 다른 소스로 판정한다.** 수집·미리보기가 서로 다른 데이터로 "완성"을 판단하면 어긋난다. **단일 권위 소스**(`['my','submissions']`) 재사용.
- **정리 실패를 화면 노출로 연결한다.** 메타를 **먼저** 비우면 Blob 삭제가 실패해도 편집 UI가 안 뜬다(순서가 안전을 만든다).
- **조회 실패에 fail-closed.** 완성 여부를 못 읽었다고 수집을 막으면 정상 사용자가 갇힌다 → fail-open + 서버 멱등 차단을 안전망으로.

## 7. 비동기 복원 루프의 stale 스냅샷 가드 (#40-B)

### 왜 필요한가? (읽어오는 동안 세상이 바뀐다)

수집 화면은 새로고침 후 IndexedDB에서 슬롯 이미지를 복원한다. 복원 루프는 **시작 시점의 슬롯 스냅샷을 순회**하며 각 슬롯마다 `await getImageBlob(...)`으로 Blob을 읽는다. 그런데 `await`는 **일시 중단점**이다 — 그 사이 사용자가 방금 복원된 슬롯을 **다른 사진으로 교체**할 수 있다. 그러면 뒤늦게 도착한 옛 복원 결과가 **새 사진의 Object URL을 덮어쓰거나**, map에서 밀어내 그 URL의 **revoke 경로를 잃게(메모리 누수)** 만든다. 이게 restore race다.

Day 10에선 이 버그가 **재현되지 않았다**(문서 §4-3). 하지만 Day 10 노트의 원칙은 **"미재현 ≠ 없음"** 이다. 재현이 어려운 경합은 실기기·특정 타이밍에서 터진다. 그래서 "관찰로 못 잡았으니 코드로 판정한다" — 경합 창이 코드에 실재하는지 추적하고, 실재하면 가드를 넣는다.

미묘한 지점: `imageKey`가 `${challengeId}:${slotIndex}`라는 **결정적 키**라서, 슬롯을 "교체"해도 **키가 그대로**다. 즉 "키가 바뀌었나?"로는 교체를 감지 못 한다. 대신 **교체가 완료되면 `imageDataUrl`이 채워진다**는 사실로 판별해야 한다. 결정적 키의 편리함(덮어쓰기 시 고아 Blob 없음)이 여기선 함정이 된다.

### Typolog에서는?

`src/features/challenge/CaptureClient.tsx`의 restore 이펙트(:104~144), 가드 :120~125:

```
const current = useChallengeStore.getState().slots.find(s => s.index === slot.index)
if (!current || current.imageKey !== slot.imageKey || current.imageDataUrl) {
  continue  // stale 복원 스킵
}
```

세 조건이 각기 다른 경합을 막는다:
- **`!current`**: 슬롯이 사라진 극단 케이스.
- **`current.imageKey !== slot.imageKey`**: 리셋 케이스. `resetDraft()`가 슬롯을 `emptySlot`(imageKey=null)으로 되돌리면 걸러진다.
- **`current.imageDataUrl`(truthy)**: **교체 케이스**. 교체는 imageKey가 같아 앞 조건을 못 통과하지만, `handleCropConfirm`의 `fillSlot`(:242)이 교체 완료 시 `imageDataUrl`(새 Object URL)을 채우므로 이 세 번째 조건이 정확히 걸러낸다.

가드가 **Object URL 생성(:127) 이전**에 위치하는 게 핵심이다 — stale 복원이 URL을 **만들고 나서** 밀려나면 그 URL은 revoke 경로를 잃고 누수된다. 가드가 앞에 있어 애초에 안 만든다. `isMounted` 플래그(:105, :141~143)는 언마운트 후 setState를 막는 별개의 표준 가드다. QA는 이 판정을 **코드 추적(경합 시나리오 3종 수기)** 으로 검증했다(자동화 테스트는 없음 — 리스크로 기록).

이건 Phase 1의 **Object URL 해제**(roadmap #10) 감각의 실전 재점검이다: "만든 URL은 반드시 revoke"인데, 경합으로 **revoke할 참조 자체를 잃는** 경로를 코드 레벨로 차단한 것.

### 비유

도서관에서 "3번 자리 책을 서고에서 꺼내오라"는 심부름(await)을 받고 서고에 다녀오는 사이, 손님이 **3번 자리 책을 스스로 새 책으로 바꿔놓았다**. 내가 뒤늦게 옛 책을 3번 자리에 꽂으면 손님의 새 책을 덮어버린다. 그래서 꽂기 **직전에** "지금 3번 자리에 이미 (새) 책이 놓여 있나?"를 확인하고, 있으면 옛 책을 그냥 반납한다(continue). 자리 번호(imageKey)는 그대로라 번호로는 구분 못 하고, **"이미 책이 놓였는지(imageDataUrl)"** 로 판별한다.

### 자주 하는 실수

- **시작 스냅샷을 그대로 믿고 await 후 반영한다.** await 창 동안 상태가 바뀔 수 있다 → 반영 **직전에 현재 상태 재확인**.
- **결정적 키로 "바뀜"을 감지하려 한다.** 교체는 같은 키라 키 비교로 못 잡는다 → `imageDataUrl` 존재 등 **다른 신호**로.
- **가드를 Object URL 생성 후에 둔다.** 이미 만든 URL이 밀려나면 revoke 참조를 잃고 누수 → 생성 **이전**에 가드.
- **"재현 안 되니 버그 아님"으로 종결한다.** 경합은 미재현이 흔하다 → 코드 레벨 경합 창 존재로 판정.

## 8. Server-Timing 헤더 계측 (#50)

### 왜 필요한가? (밖에서는 "느리다"만 보이고, 안에서 어디가 느린지는 안 보인다)

브라우저 DevTools는 요청의 **총 시간**은 보여주지만, 그 안에서 **인증이 느린지 Storage가 느린지 DB가 느린지**는 모른다. 서버 핸들러 내부는 블랙박스다. §3에서 배운 "구성 요소 분해"를 **실제 프로덕션 요청**에서도 하려면, 서버가 내부 구간 시간을 **응답에 실어 보내야** 한다. 이때 쓰는 게 표준 HTTP 헤더 **`Server-Timing`** 이다. 브라우저가 이 헤더를 파싱해 DevTools Network 탭의 Timing에 구간별로 표시해준다 — 커스텀 로깅 없이 **표준 방식**으로 내부를 들여다본다.

주의: 타이밍 수치라도 아무 요청에나 붙이면 정보가 샐 수 있다. **인증·소유권 검증을 통과한 성공 응답에만** 붙여야 한다.

### Typolog에서는?

`src/app/api/submissions/[id]/collage/route.ts`(A6 라우트):

- **구간 마킹**: `mark(name)` 헬퍼(:24~28)가 `performance.now()` 델타를 누적. 핸들러가 auth(:36) → own(:48) → form(:73) → storage(:82) → db(:101) → sign(:109) 지점에서 각각 `mark`를 찍는다.
- **헤더 조립**: 성공 직전 구간들을 `name;dur=..` 형식으로 join하고 `total`을 덧붙여 `Server-Timing` 헤더로 응답(:111~118). §3의 `profile-collage-upload.ts`(스크립트 = 라이브 라우트 밖 분해)와 **상보적** — 스크립트는 하위 레이어를, 헤더는 **실제 핸들러 내부**를 잰다(스크립트 서두 :9가 "핸들러 내부 분해는 Server-Timing 헤더가 실제 제출에서 보완"이라 명시).
- **정보 노출 없음**: 헤더는 **인증(:33~36) + 소유권(:47~51) 검증을 통과한 성공 응답에만** 실린다(:20 주석). 실패 경로(에러 응답)에는 없다. 타이밍 수치만 노출되고 경로·secret은 없다 — QA가 코드 순서로 확인.

### 비유

택배 상자에 "총 배송 3일"만 찍힌 송장 대신, **"집화 2h → 간선 18h → 배송 4h"** 처럼 구간별 시간이 찍힌 송장(Server-Timing)을 붙이는 것. 받는 사람(DevTools)이 송장만 봐도 "간선에서 지체됐구나"를 안다. 단, 이 상세 송장은 **본인 확인된 수취인(인증·소유권 통과)** 에게만 준다 — 아무에게나 주면 물류 내부가 노출된다.

### 자주 하는 실수

- **커스텀 로그로만 내부 시간을 남긴다.** 표준 `Server-Timing`을 쓰면 DevTools가 자동으로 시각화한다.
- **모든 응답에 타이밍을 붙인다.** 인증·소유권 통과 **성공 응답에만** — 실패/미인증 응답엔 안 붙여 정보 노출을 막는다.
- **경로·id 같은 값을 헤더에 담는다.** 타이밍 **수치만**. 헤더도 노출면이다.

## 9. 검수 Day의 범위 규율 — 새 범위를 얹지 않기

### 왜 필요한가? (고치다 보면 자꾸 새 일이 보인다)

검수 Day엔 코드를 파고들다 **새 개선거리·새 버그**가 계속 눈에 띈다. "이왕 만진 김에 이것도" 하다 보면 범위가 무한 팽창하고, **QA가 검증한 범위**와 **실제 커밋된 범위**가 어긋난다(검증 안 된 변경이 섞임). 그래서 검수 Day에는 **발견을 기준에 따라 분류하고, 이번 Day에서 고칠 것만 고친다.** 기준은 Day 10 결정 ⑦의 승계다: **Critical/High = 즉시 수정 / Medium = 이슈화(이관) / Low = 기록 / 신규 기능 요청 = 이슈로만.**

### Typolog에서는? (발견의 처분이 갈린 실제 사례)

- **High → 즉시 수정**: `#80`(collages UPDATE 정책 부재 — 재시도 영구 불능). 프로파일링 중 발견됐지만 High(재시도 전략을 무너뜨림)라 **이번 Day 내** 마이그레이션 0006 + verify-rls 프로브 2종으로 해소(백엔드 §9 Day10.5-(e)).
- **Medium → Phase 4 이슈화**: `M-1`(`/challenge/[id]` 진입 시 `useMySubmissions` → 챌린지 조회의 **순차 fetch** 체감 지연, QA §3 M-1). #78 리다이렉트가 낳은 새 지연이지만 기능 결함이 아닌 성능 관찰이라 **게이트 비차단** + Phase 4 이슈(#81)로.
- **Low → 기록만**: `L-1`(문서 §7 재실행 가이드가 프로브 수 36→39 미갱신), `L-2`(로직 수정 3건의 컴포넌트 테스트 부재 — 프로젝트 전반 기존 관행). QA 리포트에 남기고 끝.
- **신규 요청 → 이슈로만**: 계정 선택(#83)·오프라인/에러 UI 같은 신규 UX는 **이번 Day에 손대지 않고** 이슈만 만든다.

결과: QA 리포트(`docs/reviews/phase3-day10.5-qa-review.md`)의 **Critical 0 / High 0 / Medium 1(비차단) / Low 2**로, 커밋 범위(#74·#78·#40-B·#76·#77·#48·#50·#80)와 검증 범위가 정확히 일치한다. "고칠 것만 고쳤다"가 게이트 통과의 조건이다.

### 비유

집 수리를 부른 김에 "저 벽 색도 바꿀까, 주방도 넓힐까" 하면 공사가 끝나지 않고 검수도 못 한다. 감리(QA)가 확인한 건 "문 3개·수도 1개 수리"인데 벽까지 손대면 **검수받지 않은 변경**이 섞인다. 숙련된 감리는 발견을 **"오늘 고칠 것 / 다음에 할 것 / 기록만"** 으로 나눠, 오늘 범위를 딱 닫는다.

### 자주 하는 실수

- **"이왕 만진 김에" 새 기능을 얹는다.** 검증 범위와 커밋 범위가 어긋난다. 신규 요청은 이슈로만.
- **모든 발견을 이번 Day에 고치려 한다.** Medium/Low까지 다 고치면 Day가 안 끝나고 회귀 위험↑. 기준으로 분류해 이관.
- **High를 "나중에"로 미룬다.** 재시도 영구 불능(#80) 같은 High는 게이트 차단 사유 — 즉시 수정.
- **이관을 구두로 넘긴다.** Medium은 **이슈 번호**로 남겨야 Phase 4에서 살아난다(#81 등).

## 검수 노트 — "고치면서 더 큰 걸 발견한" 날

Day 10.5를 관통한 장면: **고치려고 판 곳에서 더 근본적인 결함이 나온다.**

- `#50` 프로파일링(A6 재측정)을 하려고 같은 path에 반복 업로드 → **`#80`(collages UPDATE 정책 부재)** 발견. 성능을 재려다 **보안·재시도 결함**을 잡았다.
- `#78`(수집 화면 정합성)을 리다이렉트로 고치니 → **`M-1`(순차 fetch 지연)** 이라는 새 성능 관찰이 생겼다(#81로 이관). 한 수정이 다른 관찰을 낳는다.

교훈: **측정·수정은 예상 못 한 결함을 드러낸다.** 그럴 때 (1) 발견을 심각도로 분류하고(§9), (2) High면 이번 Day에 회귀 프로브까지 붙여 닫고(#80), (3) 나머지는 이관해 범위를 지킨다.

## Phase 4로 가는 다리

Day 10.5로 Phase 3이 닫힌다. Phase 4(품질·관측 — PostHog·Sentry·Vitest·Playwright)로 넘어가며, 이번 Day가 남긴 이관·습관이 선행 개념이 된다.

- **오프라인/에러 UI (#40-A)**: browse에 네트워크 차단 기능이 없어 자동화 불가 → Phase 4 실기기 관찰로 이관(Day 10 §5). 오프라인 상태 UI·에러 바운더리는 **Sentry(roadmap #18)** 로 에러를 잡는 것과 짝 — "에러가 나면 사용자에게 무엇을 보여주나 + 개발자에게 어떻게 알리나".
- **계정 선택 (#83)**: 여러 계정 전환 UX는 이번 Day에 이슈로만 남겼다(§9 범위 규율). Day 7 백로그의 계정 전환 draft 누수(#53)와 이어지는 **클라이언트 상태의 사용자 경계** 주제.
- **M-1 순차 fetch 워터폴 (#81)**: `/challenge/[id]` 진입의 `useMySubmissions` → 챌린지 조회 순차 체인. 두 쿼리 병렬화(로딩 상태 합성) 리팩터링은 **성능 기준선에 콜드 로드 실측 추가**(Phase 4 measure-perf 확장)와 함께 다룰 과제.
- **운영 cleanup 잡**: Storage 고아는 현재 0건(일회성 정리 불필요)이지만, 지속 운영에선 **정기 cleanup 잡**이 필요하다(Day 10 §"다리" #40-D). Storage+DB 비원자성(Phase 2 Day 3~4)이 낳는 고아를 주기적으로 청소하는 배치 — 관측(OpenTelemetry 부록)과 연결.
- **Vitest 습관 확장 (roadmap #19)**: 이번 Day의 **red→green(테스트 우선)** 과 **워커 풀·재시도 시나리오 단위 테스트**가 Phase 4 Vitest 본격화의 예고편이다. `use-submission.test.ts`는 프로젝트 최초의 **React Testing Library(renderHook)** 사용 — Phase 4에서 컴포넌트 테스트(FeedCard·LetterGrid)로 확장되고, L-2(로직 수정의 회귀 테스트 부재)가 그 대상이다.

비유: Day 10.5가 **하자 보수를 끝내고 준공 승인 도장을 받은 날**이라면, Phase 4는 **입주 후 하자 접수 창구(Sentry)·CCTV(PostHog)·정기 안전 점검(Vitest·Playwright)을 설치하는 날**이다. 보수가 깨끗할수록(Critical/High 0) 창구 개설이 순조롭다.

## 핵심 한 장 요약

- **테스트 우선(#74)**: 1줄 수정일수록 red→green. `use-submission.test.ts`가 3키 계약(submission·my·**feed**)을 고정 후 `use-submission.ts:62` 한 줄. `['feed']` prefix가 하위 키 포괄. 값 vs 멤버십 판단.
- **RLS CRUD 대칭(#80)**: upsert=UPDATE. collages에 UPDATE 정책 부재 → 재시도 영구 불능(High). 0006으로 `collages_update` 추가(letter_pieces_update와 동형), verify-rls 허용+차단 짝 프로브. "매번 새 경로"가 사각지대였음.
- **단일 표본 금지(#50)**: A6 10.6s(단일 표본) → `profile-collage-upload.ts` 반복 측정+구성 분해(인증/Storage/DB/서명) → 웜 p50 164ms = **이상치**. 측정이 병렬화 착수를 결정.
- **워커 풀(#50)**: cap 3개 워커가 공유 큐 `shift()` 소비(JS 단일 스레드라 원자적). 실패분만 1회 재시도, 재실패는 전파(전체 재실행 안전망). 진행 표시 순번→완료 누적(k/N).
- **경합 안전 멱등(#48)**: SELECT-후-INSERT의 TOCTOU를 UNIQUE로 직렬화. onConflictDoNothing+returning 0행=중복 감지→REPORT_ALREADY_EXISTS. 선정리 DELETE+제약을 같은 트랜잭션에, `(created_at,id)` 결정적 타이브레이크.
- **화면 정합성(#78)**: 진입로 차단(CaptureClient 리다이렉트) + 잔존 draft 정리(CollagePreviewClient) 짝. `['my','submissions']` 단일 소스 재사용. fail-open + 서버 멱등 차단.
- **stale 스냅샷 가드(#40-B)**: restore await 창의 교체 경합을 반영 직전 현재 상태 재확인으로 차단. 결정적 imageKey라 교체 감지는 `imageDataUrl` 존재로. "미재현 ≠ 없음" 코드 판정.
- **Server-Timing(#50)**: 핸들러 구간(auth/own/form/storage/db/sign/total)을 표준 헤더로 노출. 인증·소유권 통과 성공 응답에만.
- **범위 규율(#9)**: High(#80) 즉시 수정 / Medium(#81) 이관 / Low 기록 / 신규(#83) 이슈만. 검증 범위 = 커밋 범위. Critical/High 0.
