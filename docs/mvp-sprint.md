## 1. MVP 핵심 가설

### 가설 1 — 완성 경험이 공유 욕구를 만든다

> "사용자가 일상에서 글자를 직접 찾아 crop하고 콜라주를 완성하면,
> 결과물을 누군가에게 보여주고 싶어진다."

검증 방법: 콜라주 완성 후 공유 버튼 클릭률 20% 이상
이것이 증명되면: 피드/비교 기능을 다음 단계로 진행
이것이 실패하면: 완성 경험 자체(crop UX, 콜라주 비주얼)를 먼저 개선, 또는 문장 난이도 조정

### 가설 2 — 같은 문장, 다른 결과가 비교의 재미를 만든다

> "공유된 콜라주를 본 사람이 '나도 해보고 싶다'고 느낀다."

검증 방법: 공유 페이지(/s/[id]) 접속 후 "나도 만들기" 클릭률 15% 이상
이것이 증명되면: 바이럴 루프가 작동함 — 소셜 피드 확장 진행
이것이 실패하면: 공유 페이지 디자인 개선, 또는 비교 진입점 강화

## 4. 챌린지 문장 정책 + 첫 10개 문장 후보

### 문장 길이 기준

- 공백 제외 6~8글자 (기존 6~10글자에서 상한 낮춤)
- 제외 이유: 10글자는 솔로 첫 사용 경험에서 너무 많다. 6~8글자면 15~25분 안에 완성 가능. 재미를 느끼기 전에 지치는 것을 방지한다.
- 첫 2주는 6~7글자 위주로 운영, 익숙해지면 8글자로 올린다.

### 문장 선정 기준

1. **일상 밀착성**: 동네, 카페, 지하철, 편의점 등 일상 동선에서 글자를 찾을 수 있어야 한다
2. **감성 공명**: 읽었을 때 "이 문장 좋다"는 느낌. 콜라주 완성 후 SNS에 올릴 때 캡션이 될 수 있어야 한다
3. **완성 가능성**: 희귀한 글자(ㅋ, ㄳ 등)나 특수한 문자 포함 금지. 흔한 한글 자모 조합
4. **중의성 없음**: 해석이 하나여야 한다. 낯선 신조어, 비속어 금지
5. **공백 포함 표기 허용**: 문장 내 공백은 슬롯에 포함하지 않음 (예: "오늘 참 좋다" → 슬롯 6개)

### 공백/특수문자 처리 규칙

- 공백: 슬롯 미포함. 콜라주 줄 배치는 작성자 지정 줄 배열(`Challenge.lines`)을 따름 (자동 줄바꿈 아님 — 아래 "줄 배치" 참고)
- 느낌표(!), 물음표(?): MVP에서 제외. 글자 슬롯 복잡도 증가
- 마침표, 쉼표: 제외
- 숫자: 제외 (한글만)
- 영문: 제외 (한글만)

### 첫 10개 문장 후보

| # | 문장 | 공백 제외 글자 수 | 난이도 | 선정 이유 |
|---|------|-----------------|--------|----------|
| 1 | 오늘도 화이팅 | 6 | 쉬움 | 첫 챌린지. 부담 없는 시작. "오늘", "화", "팅" 모두 주변에서 찾기 쉬움 |

## 5. 핵심 Funnel

MVP에서 측정할 퍼널. PostHog은 배포 직후 추가하지만, 수동 확인으로도 추적 가능한 구조로 설계한다.

### 퍼널 1: 챌린지 완성 퍼널 (가장 중요)

```
홈 화면 접속 (100%)
  → 챌린지 시작 클릭 (목표: 50%)
    → 첫 글자 crop 완료 (목표: 70%)
      → 전체 글자 완성 (목표: 45%)
        → 콜라주 PNG 생성 (목표: 85%)
          → 제출 (목표: 60%)
```

목표 전환율을 기존 product-brief보다 낮게 잡은 이유:
- 솔로 베타 10~20명 규모에서는 완벽한 퍼널보다 이탈 구간 발견이 목적
- "시작 → 첫 글자" 70%가 핵심: 이 구간이 낮으면 crop UX 문제

### 퍼널 2: 공유 바이럴 퍼널

```
제출 완료 (100%)
  → 공유 링크 생성/복사 (목표: 25%)
    → 공유 페이지 조회 (목표: 링크 복사의 40%)
      → "나도 만들기" 클릭 (목표: 공유 페이지 조회의 15%)
```

이 퍼널에서 "나도 만들기" 클릭이 발생한다면 가설 2 검증 성공.

### 2주 MVP 성공 판단 기준

| 지표 | 최소 성공 기준 |
|------|-------------|
| 콜라주 완성 건수 | 베타 사용자 10명 중 8명 이상 1회 이상 완성 |
| 공유 링크 생성 | 완성 건수의 20% 이상 |
| 바이럴 유입 | 공유 링크 통해 신규 접속 1건 이상 |
| 크리티컬 버그 | 0건 (카메라 접근, crop, 저장 불가 등) |
| 사용자 정성 피드백 | "재밌다", "공유하고 싶다" 반응 5건 이상 |

## 7. 제품 리스크

### 리스크 1 — Crop UX가 모바일에서 너무 어려움 (해소됨)

설명: 핀치 줌 + 드래그 crop을 Canvas로 직접 구현하면 iOS Safari에서 터치 이벤트 충돌이 자주 발생한다. Day 4 일정을 초과할 가능성 가장 높았던 리스크.

대응 결과 (Day 4 완료):
- react-easy-crop으로 먼저 구현 → 실기기 테스트에서 "고정 정사각형 + 이미지 이동" 방식이 정밀 crop에 부적합 판명
- react-image-crop(자유 영역 그리기) 으로 교체하여 채택. "고정 사각형 단순화" fallback은 불필요해져 기각
- 직접 구현 대신 라이브러리 활용으로 일정 내 해결

### 리스크 2 — iOS Safari에서 카메라/저장 동작 불일치

설명: iOS Safari는 카메라 접근, blob URL 다운로드, Web Share API 동작이 Android Chrome과 다르다. 테스트 없이 배포하면 iOS 사용자 이탈 폭증.

대응 방안:
- Day 3부터 실제 iPhone으로 매일 테스트 (시뮬레이터 불충분)
- PNG 저장은 `<a download>` 대신 새 탭에서 이미지 열기 방식 병행
- iOS 전용 분기 처리를 `src/lib/platform.ts`에 중앙화

### 리스크 3 — 콜라주 완성까지 시간이 너무 걸려 이탈

설명: 6~8글자를 각각 촬영 + crop하면 실제로 15~30분이 소요된다. 처음 사용자는 중간에 포기할 가능성이 높다.

대응 방안:
- 첫 주는 6글자 문장으로만 운영
- "이어하기" 기능(M-13) 반드시 Day 5에 완성 (세션 간 진행 복원)
- 온보딩에서 "한 글자씩 천천히, 오늘 하루 동안 모아도 됩니다" 명시
- 슬롯별 crop 시간을 나중에 PostHog으로 분석해 개선 포인트 파악

### 리스크 4 — Supabase RLS 설정 실수로 데이터 노출

설명: RLS(Row Level Security)를 잘못 설정하면 다른 사용자의 비공개 제출물이 보일 수 있다. 솔로 개발이라 리뷰 없이 배포하면 위험.

대응 방안:
- Day 8에 RLS 정책 작성 후 별도 익명 세션으로 직접 접근 테스트
- 기본 정책: 모든 테이블 `SELECT` 는 `is_public = true` 또는 `auth.uid() = user_id`
- `letter_pieces` 버킷: 업로드는 인증 사용자만, 조회는 본인만
- Day 13 배포 전 RLS 체크리스트 재확인

### 리스크 5 — 일정 지연 시 범위 축소 기준 불명확

설명: Day 10~11에 Supabase 연동이 막히면 Day 14 베타 배포가 불가능해질 수 있다. 솔로 개발에서 막히면 전체가 멈춘다.

대응 방안:
- 긴급 Fallback 플랜 — Day 12까지 서버 연동이 안 되면:
  - PNG 다운로드 + 개인 공유 (링크 없이 이미지 파일만)로 베타 진행
  - 서버 연동은 베타 이후 Week 3으로 연기
  - 이 경우에도 "완성 경험이 공유 욕구를 만드는가" 가설은 검증 가능
- 매일 Day 종료 시 완료 기준 체크. 2개 이상 미완이면 다음 날 범위 조정
