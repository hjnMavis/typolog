## 2. Invalidation Map (실증)

| Mutation | `['feed', cid]` | `['my','submissions']` | 실측 결과 |
|---|---|---|---|
| S1 좋아요 | setQueryData 낙관 ±1 → onSuccess 권위값 정정 (invalidate 없음) | — | ✅ 낙관 1 즉시 → 서버 rc=1 → 취소 → rc=0 |
| S4 공개 토글 | **invalidate** (멤버십 변화) | setQueryData 단일 정정 (값 변화) | ✅ 배지 즉시 전환(대상 카드만), 피드 재조회 시 등장/소멸 |
| S3 닉네임 | **invalidate** (카드에 닉네임) | — (헤더는 별도 소스) | ✅ 저장 → 피드 카드 닉네임 즉시 반영 |
| 제출 완성 (A4) | **invalidate** (멤버십 변화 — V-1 수정, Day 10.5 #74) | invalidate | ✅ 유닛 테스트로 고정 (`tests/unit/use-submission.test.ts`) |
| 로그아웃 | 전체 clear | ← | ✅ 로그아웃 → `/login`, 보호 라우트 재접근 차단, 개인 데이터 잔존 없음 |

**캐시의 사용자 국소성 (크로스 유저 반영 타이밍)** — 실측으로 확정:

- A가 비공개 토글 → **A의 invalidate는 A 브라우저만** 비운다.
- B의 무조작 화면: 마플 카드 **잔존** (16:59 관찰 — 정상, "느림")
- B의 서버 fetch: **즉시 제외** (서버 진실)
- B의 재진입(재검증 트리거): **소멸** (정상)
- `/s`·OG는 서버 판정이라 **토글 즉시 404/200** — 캐시 지연 없음
- **판정 기준선**: B 화면의 잔존은 staleTime(60s)+다음 트리거(재진입·포커스)까지 = 정상. 재진입 후에도 잔존하면 고장.
